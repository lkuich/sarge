# Sarge

Sarge is a lightweight event pixel and ingestion API for debugging media and product tracking. It records first-party events, observes common third-party pixel calls, and can mark project-scoped test traffic so teams can inspect user flows without polluting real production traffic. The codebase is a TypeScript `pnpm` workspace with a browser pixel, an Express API, Prisma, Postgres, and one root Docker Compose setup for local development.

Projects are split into persisted environments: Production, Staging, and Development. Each environment has its own pixel URL, tracking endpoint identity, event stream, user/session flow view, debug stream, event hosts, webhooks, and public verify URL. AI recommendations currently run for Production; Staging and Development keep the same tracking surfaces but show AI review as disabled.

## Quick Start

Install dependencies:

```bash
pnpm install
```

Start the local database and API:

```bash
docker compose up postgres api
```

Run Prisma migrations against the Compose database:

```bash
DATABASE_URL="postgresql://sarge:sarge@localhost:5432/sarge_dev?schema=public" pnpm prisma:migrate
```

Run local checks:

```bash
pnpm test
pnpm typecheck
pnpm build
```

## Browser Pixel

The v2 browser API is intentionally small:

```html
<script src="/index.global.js"></script>
<script>
  sarge('init', {
    siteId: 'env_123_production',
    endpoint: 'http://localhost:49828',
    attributionTtlDays: 28
  });

  sarge('track', 'Purchase', {
    value: 129.99,
    currency: 'USD',
    orderId: '1234'
  });
</script>
```

The pixel reads `sarge_ref` and `sarge_aff` URL parameters, stores attribution in `localStorage`, and sends events to the API with `sendBeacon`, `fetch(..., { keepalive: true })`, or a compact image GET fallback. The browser/API field is still named `siteId`, but hosted installs should pass the selected environment ID, such as `env_123_production`, `env_123_staging`, or `env_123_development`.

Hosted snippets use environment-specific pixel URLs:

```html
<script async src="https://track.sargetrack.app/pixel.js?env={siteEnvironmentId}"></script>
```

### Test Traffic and Impersonation

After the hosted pixel loads, testers can run these page-console commands:

```js
impersonate('customer_123');
clear_impersonation();
```

`impersonate()` makes future browser events appear under the target `userId` while marking them as Sarge test traffic. The marker is stored in event properties:

```json
{
  "sarge_test": true,
  "sarge_test_mode": "impersonation",
  "sarge_tester_user_id": "project-scoped-tester-id",
  "sarge_impersonated_user_id": "customer_123"
}
```

The Sarge console can hide real/test traffic, show test traffic only, and distinguish real `customer_123` traffic from events generated by an impersonating tester. This is a debugging tool only; it does not grant application auth or permissions.

## API

Health:

```http
GET /healthz
```

Primary ingestion:

```http
POST /v2/events
Content-Type: application/json

{
  "siteId": "env_123_production",
  "name": "Purchase",
  "occurredAt": "2026-06-19T12:00:00.000Z",
  "sessionId": "sess_123",
  "userId": "user_123",
  "properties": {
    "value": 129.99,
    "currency": "USD"
  }
}
```

Compact fallback:

```http
GET /v2/e?sid=env_123_production&n=Lead&ts=2026-06-19T12:00:00.000Z&ss=sess_123&u=user_123
```

Server-side ingestion for trusted backend events:

```http
POST /v2/server/events
Authorization: Bearer sarge_sk_...
Content-Type: application/json

{
  "siteId": "env_123_production",
  "name": "purchase.completed",
  "eventId": "order_123",
  "occurredAt": "2026-06-24T12:00:00.000Z",
  "userId": "customer_123",
  "properties": {
    "order_id": "order_123",
    "value": 129.99,
    "currency": "USD"
  }
}
```

Use this for backend facts such as paid orders, refunds, subscriptions, fulfillment events, and payment/webhook confirmations. Server events are authenticated against `SiteEnvironment.serverEventSecretHash`, normalized into the same `Event` table, and stored with `source: "server"`.

Tokenized postback URLs for affiliate networks and partner tools:

```http
GET /v2/postback/env_123_production/postback_token_...?event=affiliate.conversion&click_id=click_123&order_id=order_123&value=42.50&currency=USD
```

Use postback URLs when the sender cannot set custom headers. The URL token is checked against `SiteEnvironment.postbackTokenHash`; accepted events are stored with `source: "postback"`. Prefer `/v2/server/events` whenever the sender supports headers, because URL tokens are more likely to leak through logs, screenshots, or copied partner setup links.

Sarge stores server/postback credentials as SHA-256 hashes:

```bash
node -e "const { createHash, randomBytes } = require('node:crypto'); const token = 'sarge_sk_' + randomBytes(32).toString('base64url'); console.log('token=' + token); console.log('sha256=' + createHash('sha256').update(token).digest('hex'));"
```

Set the generated hash on `SiteEnvironment.serverEventSecretHash` for `/v2/server/events` or `SiteEnvironment.postbackTokenHash` for `/v2/postback/...`.

## Distribution

Sarge has three planned distribution paths:

- self-hosted: customer runs the API/database and hosts the pixel
- hosted shared: Sarge provisions a custom endpoint and customized pixel on shared infrastructure
- dedicated hosted: "Contact us" managed infrastructure for teams that need isolation or custom requirements

## Product Direction

Sarge is being repositioned as a cheap, dead-simple event recorder for teams that need to debug tracking implementations. The product direction includes real-time event streams, a third-party pixel watchdog, test-traffic filters, custom script deployment, deterministic AI reviews, and an LLM query layer over structured event data.

See `docs/STRATEGY.md`.
